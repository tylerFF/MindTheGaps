## Three Concerns Explicitly Addressed

### 1. Will JotForm submissions reach HubSpot? ✅ YES

The same `mtg-quiz-webhook` worker handles both HTML and JotForm submissions. It calls `createHubSpotClient(env.HUBSPOT_API_KEY).upsertContact(email, hubspotProps)` regardless of source (`src/index.js` line 361). Contact dedupes by email. All `mtg_` properties (primary gap, score, sub-diagnosis, cost-of-leak, eligibility, tracking params) get written the same way.

**Evidence:** The JotForm has 31 existing submissions — if HubSpot writes were broken, Marc would have noticed by now. We'll still verify this in the end-to-end test.

### 2. Will the scan prefill URL work so Marc doesn't re-type contact info? ✅ YES

When the webhook processes a quiz submission, it generates `mtg_scan_prefill_url` (`src/index.js` line 271, built by `buildScanPrefillUrl` in `quizEmail.js` line 270-282). This URL is stored on the HubSpot contact. When Marc clicks it, the scan JotForm opens with these fields prefilled:
- `scanFirstName` ← quiz firstName
- `contactEmail` ← quiz email
- `scanBusinessName` ← quiz businessName
- `scanIndustry` ← quiz industry
- `quizPrimaryGap` + `primaryGap` ← scoring result

Marc only has to fill in scan-specific fields (baselines, actions, etc.) — not contact info.

**Why this works regardless of quiz source:** The prefill URL is generated by the worker from the extracted payload. Whether that payload came from custom HTML or JotForm, the same contact fields get extracted and the same URL gets built.

### 3. Will users see the same results page with the scan booking CTA? ✅ YES

Both quiz paths converge on the same `/results/` page at `pages/results/index.html`. The page renders:
- Primary gap statement + baseline score badge
- Sub-diagnosis + key signals
- Cost-of-leak + advice
- Fastest next steps
- **If eligible:** "Book the 45-Minute Growth Gap Scan — CAD $295" button → Stripe checkout (`STRIPE_CHECKOUT_URL` constant on line 423)
- **If not eligible:** "Before We Can Book Your Scan" guidance

The HTML quiz currently redirects directly to `/results/#ENCODED_DATA`. The JotForm path redirects to `/loading/?sid=X`, which polls for the same `resultsUrl` and redirects the user there. **Same final page, same CTAs, same everything.**

### UX nuance to be aware of

The JotForm path adds a brief (~1-2 second) visual flash of JotForm's native "Thank You!" page before the auto-redirect to the loading page. This is a JotForm behavior we can't fully eliminate. We can minimize it by configuring JotForm's thank-you page to be a minimal spinner that matches our loading page styling — but this is cosmetic, not functional. Add to verification: time the full JotForm → results flow and confirm it's acceptable.
